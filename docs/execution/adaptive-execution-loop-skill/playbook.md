# playbook

## default execution loop
- define the terminal state before experiments begin
- create the task folder before the first uncertain execution step
- log each meaningful experiment with evidence and a next decision
- keep `task.md` current as the synthesized operating view
- promote only durable wins into the playbook

## stable tactics
- separate the reusable skill from repo-local execution memory
- prefer a small deterministic script pair over heavy tracker infrastructure
- keep experiments as truth and `task.md` as synthesis

## failure patterns
- repeating a failed method without a changed hypothesis
- splitting the same idea across multiple drifting tracker files
- treating the experiment log like a diary instead of a decision tool

## verification patterns
- validate the skill package itself, not just the copied task files
- prove helper scripts with dry-run plus real execution
- verify idempotence before calling the tracker workflow reliable
- regenerate and validate metadata from the final skill text before calling the package complete

## promotion rules
- promote only evidence-backed lessons that should change the default method next time
